# Project Architecture — Blog (Hexagonal)

This document defines the project's directory structure, following **Hexagonal
Architecture (Ports & Adapters)**. The initial implementation covers only the
**GraphQL** input adapter, but the application core's `ports` are designed to
support new adapters (gRPC, HTTP/REST, WebSocket, CLI, etc.) without changes to
the domain or the use cases.

Since this is a study project, there's no concern about external consumers or
multiple teams — the structure prioritizes didactic clarity and the ability to
add/remove an adapter as a whole.

## General principle

```
Outside world → Adapter (in) → Port (driving) → Usecase → Port (driven) → Adapter (out) → Outside world
```

The core (`internal/app`) never imports anything from `internal/adapters`. The
dependency always points inward.

Each input adapter is a **self-contained, deletable** unit: contract
(schema/proto), generated code, implementation, and mapping all live together
in the same directory. There's no shared `/api` at the root — if an adapter is
removed, nothing is left orphaned outside of it.

Infrastructure concerns (configuration loading, opening a database connection,
running migrations) are **not** adapters — they don't implement any port. They
live in a separate `internal/infra` layer and are wired together in `main.go`,
which then injects the ready-to-use resources into the adapters:

```
infra (config, connections) ──┐
                               ├──> injected by main.go
adapters (in / out) ──────────┘
                    │
                    ▼
                  ports
                    │
                    ▼
                  usecase
                    │
                    ▼
                  domain
```

## Layer descriptions

| Layer | Responsibility | May import from |
|---|---|---|
| `domain` | Entities, invariants, and pure business rules | nothing (stdlib only) |
| `usecase` | Orchestrates business rules, implements the driving port, depends only on the driven port | `domain`, `ports` |
| `ports` | Contracts (interfaces) between the core and the outside world | `domain` |
| `infra` | Loads configuration and opens shared resources (DB connections, etc.); does not implement any port | nothing from the domain |
| `adapters/in` | Translates an external protocol (GraphQL, and later gRPC/HTTP/WS/CLI) into calls to the driving port | `ports`, `domain` (return types only) |
| `adapters/out` | Implements the driven port (persistence, etc.), receiving already-open resources from `infra` | `ports`, `domain` |
| `pkg` | Reusable utility implementations (logger, etc.) | nothing from the domain |

### Why `infra` is separate from `adapters/out`

- `infra/database/postgres.go` only knows how to open/close a connection and run
  migrations. It has no knowledge of `domain.Post` and does not implement
  `Repository`.
- `adapters/out/db/postgres/post.go` implements the `repository.go` port, but
  **receives the ready-made connection as a dependency** (constructor
  injection) instead of creating it itself.

This keeps the "delete the whole adapter" test valid: swapping Postgres for
Mongo means swapping `adapters/out/db/postgres` entirely, and `infra/database`
changes too (it's driver-specific), but `infra/config` stays the same — it only
describes *what* to configure (DSN, port, etc.), not *how* to connect.

`cmd/server/main.go` is what wires everything together: load config → open the
connection via `infra/database` → inject the connection into the output
adapter → inject the output adapter into the usecase → inject the usecase into
the GraphQL resolver.

## Golden rule

No file inside `internal/app` may import anything from `internal/adapters` or
`internal/infra`. If that happens, it's a sign the hexagonal architecture has
been violated.

## Pattern for new input adapters

When adding a new protocol, create a sibling directory to `graphql` inside
`internal/adapters/in/`, following the same self-containment logic:

```
/internal/adapters/in/<protocol>
├── <contract>               # .proto, routes, commands, etc. (if applicable)
├── /generated                # generated code, if the protocol uses codegen
├── /server or /handler       # implementation that calls the driving port
└── /mapper                   # domain <-> protocol model
```

Validation test: removing the entire adapter directory should require no
changes to `internal/app`.

## Extension roadmap

- [x] GraphQL (`adapters/in/graphql`) + in-memory repository (`adapters/out/db/memory`)
- [ ] Postgres repository (`adapters/out/db/postgres`) + `infra/database/postgres.go` connection setup
- [ ] gRPC adapter (`adapters/in/grpc`) — reusing the same use cases
- [ ] HTTP/REST adapter (`adapters/in/http`)
- [ ] WebSocket adapter (`adapters/in/websocket`)
- [ ] CLI adapter (`adapters/in/cli`)
- [ ] Cache (Redis) as a repository decorator
- [ ] Messaging / domain events (`domain/event.go` + `ports/eventpublisher`)
